## Phase 1 — P0 blockers (target: 4 commits, ~4–6 hours)

### Commit 1 — F12: HRV reads from raw Polar RR, not from sync output

**Cause**: `pipeline.py::run_analysis` calls `compute_hrv_features(cleaned_synced)`. The merge with 1 Hz EmotiBit decimates the 60–80 Hz heartbeat-level Polar RR stream into 1 Hz, losing 21% of beats at a normal adult heart rate (74 bpm on s05) and inflating RMSSD by 29.8%.

**Fix**: split the pipeline so HRV reads the raw Polar DataFrame. Sync stays as-is for EDA / accel / timeseries-plot alignment.

```python
# pipeline.py (after)
def run_analysis(emotibit_df, polar_df):
    synced = synchronize_signals(emotibit_df, polar_df)
    cleaned, motion_ratio = clean_signals(synced)

    # HRV features: computed from RAW polar (beat-level resolution)
    rmssd, sdnn, mean_hr, rr_source = compute_hrv_features(polar_df)
    freq = compute_hrv_frequency_features(polar_df, resample_hz=4.0)

    # EDA + coupling features: computed from synced (need time alignment)
    eda_mean, eda_phasic = compute_eda_features(cleaned)
    ...
```

**Test**: `tests/test_pipeline_real.py::test_welltory_s05_rmssd` asserts `abs(pipeline.rmssd_ms - 35.28) < 1.0` using the committed `data/samples/welltory/welltory_s05_polar.csv`.

**Risk**: loses movement-artifact filtering on HRV (currently applied via `clean_signals` on accel). Documented in the quality-flag text; full artifact-aware HRV filtering deferred to a later enhancement.

### Commit 2 — F1: drop derived `rr_ms` write in `sync.py`

**Cause**: `sync.py:64` writes `merged["rr_ms"] = 60000 / hr_bpm` when the Polar input lacks native RR. `features.py::compute_hrv_features` then sees `rr_ms` in columns and reports `rr_source: native_polar`. Quality flag reads "research-grade" for what is actually arithmetic-derived.

**Fix**: remove the derived-RR write. Let the synced dataframe keep `rr_ms` only when native RR was present on the input. After Commit 1, HRV runs on raw Polar anyway, so the synced rr_ms column is needed only for timeseries plots — where deriving it on read is fine and can be labelled honestly.

### Commit 3 — F2 + F6: minimum-sample guard on `/api/v1/analyze`

**Cause**: empty CSV → 200 with `stress_score: 0.5`, NumPy `RuntimeWarning: Mean of empty slice`. 1-row CSV → 200 with `rmssd_ms: 0.0`. Both lie to clients that trust HTTP 200.

**Fix**: add a minimum-sample check at the top of `run_analysis` (after parsing). Throw a custom `InsufficientDataError` with a structured payload; the route handler converts to 422.

```python
# pipeline.py
class InsufficientDataError(ValueError):
    def __init__(self, detail: str, n_polar: int, n_emotibit: int):
        super().__init__(detail)
        self.detail = detail
        self.n_polar = n_polar
        self.n_emotibit = n_emotibit

MIN_BEATS_FOR_HRV = 50        # Kubios minimum for RMSSD stability
MIN_SAMPLES_FOR_SYNC = 30     # 30 s at 1 Hz minimum session

def run_analysis(emo, polar):
    if len(polar) < MIN_BEATS_FOR_HRV:
        raise InsufficientDataError(
            f"need ≥ {MIN_BEATS_FOR_HRV} beats for HRV analysis; got {len(polar)}",
            n_polar=len(polar), n_emotibit=len(emo))
    if len(emo) < MIN_SAMPLES_FOR_SYNC:
        raise InsufficientDataError(
            f"need ≥ {MIN_SAMPLES_FOR_SYNC} EmotiBit samples; got {len(emo)}",
            n_polar=len(polar), n_emotibit=len(emo))
    ...
```

**Minimum-beat basis**: Kubios HRV user guide recommends ≥ 50 beats for RMSSD stability; Task Force (1996) guideline is 256 beats for short-term HRV frequency-domain. 50 beats is the conservative-but-not-paralysing minimum for time-domain features.

**Reference**: Task Force of the European Society of Cardiology and the North American Society of Pacing and Electrophysiology. (1996). Heart rate variability. *Circulation*, 93(5), 1043–1065. https://doi.org/10.1161/01.CIR.93.5.1043

**Test**: empty-CSV POST returns 422 with detail `"need ≥ 50 beats..."`; 1-row CSV returns 422; 50-beat CSV succeeds; 49-beat CSV returns 422.
